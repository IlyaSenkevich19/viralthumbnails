# Manual Payments -> Credits (MVP Plan)

Документ для сценария, где оплата идет по прямой ссылке Stripe/посредника, а подтверждение приходит на email.
Цель: безопасно начислять кредиты пользователю без полноценного Stripe webhook на старте.

---

## Контекст

- Сейчас нет прямой интеграции Stripe webhook в продукт.
- Платежи подтверждаются письмами на почту.
- Нужно автоматически/полуавтоматически начислять кредиты в приложении по email покупателя.

---

## Короткий ответ: реализуемо?

Да. Реализуемо в формате MVP.

Лучше делать не "вебхук из почты" напрямую в БД, а через контролируемый backend endpoint с проверками:

- аутентификация источника;
- идемпотентность по `payment_id`;
- валидация плана/суммы;
- журнал начислений (ledger/audit).

---

## Рекомендуемая архитектура MVP

1. Оплата -> уведомление приходит на Gmail.
2. Автоматизация (Apps Script / Make / Zapier) извлекает:
   - email покупателя;
   - external payment id;
   - оплаченный пакет/сумму;
   - timestamp.
3. Автоматизация вызывает backend endpoint:
   - `POST /api/billing/manual-credit`
4. Backend:
   - ищет пользователя по email;
   - вычисляет количество кредитов по таблице соответствий;
   - выполняет транзакцию начисления;
   - сохраняет запись в ledger;
   - блокирует повторное начисление по `external_payment_id`.

---

## Минимальные требования к backend endpoint

## `POST /api/billing/manual-credit`

### Request (пример)

```json
{
  "email": "buyer@example.com",
  "external_payment_id": "pay_12345",
  "plan_code": "starter_100",
  "amount": 19.99,
  "currency": "USD",
  "source": "gmail_mediator_v1",
  "paid_at": "2026-04-13T12:30:00Z"
}
```

### Валидации

- `email` обязателен, нормализованный.
- `external_payment_id` обязателен и уникален.
- `plan_code` должен быть в allowlist.
- подпись/секрет в заголовке обязателен (например, `X-Manual-Billing-Signature`).

### Response (пример)

```json
{
  "ok": true,
  "user_id": "uuid",
  "credited": 100,
  "new_balance": 245
}
```

---

## Идемпотентность (критично)

Нужно хранить отдельную таблицу, например `billing_events`:

- `external_payment_id` (unique index)
- `email`
- `plan_code`
- `credits_applied`
- `status` (`applied`, `duplicate`, `rejected`)
- `payload_json`
- `created_at`

Если `external_payment_id` уже встречался -> вернуть `ok: true` + `duplicate: true`, не начислять повторно.

---

## Маппинг "оплата -> кредиты"

Нужна явная таблица:

- `starter_100` -> 100 кредитов
- `pro_500` -> 500 кредитов
- и т.д.

Не вычислять кредиты "из суммы" на лету без фиксированного соответствия (избежим ошибок при скидках/налогах).

---

## Автоматизация из почты (варианты)

## Вариант A: Gmail + Apps Script

- триггер по новым письмам;
- парсинг письма;
- вызов backend endpoint.

Плюсы: дешево, быстро.
Минусы: хрупкость парсинга шаблона письма.

## Вариант B: Make/Zapier

- фильтры, retry, визуальные сценарии;
- меньше кода, больше control panel.

---

## Безопасность

- Секрет для автоматики (`BILLING_AUTOMATION_SECRET`).
- IP allowlist (если доступно).
- Rate-limit на endpoint.
- Логи всех попыток (успешных и ошибочных).

---

## Операционный режим (рекомендуется на старте)

- Полуавтомат:
  - письмо -> запись в Google Sheet "pending";
  - ручное подтверждение (`approved=true`);
  - только после этого endpoint начисления.

Так проще ловить ошибки в начале и не терять деньги/кредиты.

---

## План внедрения

1. Добавить backend endpoint + таблицу `billing_events` + транзакционное начисление.
2. Добавить admin-лог для просмотра операций начисления.
3. Подключить автоматизацию из почты (Apps Script или Make).
4. Прогнать тесты:
   - happy path;
   - duplicate payment id;
   - email не найден;
   - plan code invalid.
5. Ввести ручной fallback: "начислить по email из админки".

---

## Что позже заменить полноценным Stripe

- Источник событий (email parsing) -> Stripe webhook.
- Вся остальная схема (идемпотентность, ledger, mapping, audit) остается актуальной.
