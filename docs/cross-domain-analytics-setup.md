# Настройка сквозной аналитики между доменами (GA4 + GTM + Google Ads)

План для сценария, когда лендинг и приложение находятся в разных репозиториях и на разных доменах.
Цель: одна аналитическая воронка и корректная атрибуция между доменами.

---

## Что должно быть единым

- Один `GA4` property.
- Один `Google Tag Manager` container (рекомендуется для старта).
- Один набор конверсий `Google Ads` (без дублей источников).

Это полностью совместимо с архитектурой "2 домена / 2 репозитория".

---

## 1) Базовая схема

1. Подключить один и тот же `GTM-XXXXXXX` на оба сайта.
2. Через GTM развернуть `GA4 Configuration` tag на обоих доменах.
3. В GA4 включить cross-domain measurement для обоих доменов.
4. Настроить передачу `utm_*`, `gclid`, `fbclid` между доменами.
5. Настроить конверсии в Ads (через импорт из GA4 или через Ads Tag).

---

## 2) Настройка GA4 (cross-domain)

В GA4:

- `Admin` -> `Data Streams` -> нужный web stream.
- `Configure tag settings` -> `Configure your domains`.
- Добавить оба домена (лендинг и приложение).
- Проверить, что переходы между доменами не создают новую сессию (через Realtime/DebugView).

Пример доменов:

- `landing.example.com`
- `app.example.com`

---

## 3) Настройка GTM

1. Один контейнер установлен на оба сайта.
2. `GA4 Configuration` tag:
   - Measurement ID одинаковый для двух доменов.
   - Trigger: All Pages.
3. Включить linker/cross-domain в конфиге GA4 тега (или через fields/settings, в зависимости от шаблона).
4. Проверить в GTM Preview:
   - Page view на лендинге.
   - Клик на CTA.
   - Переход в приложение.
   - События регистрации/активации.

---

## 4) Google Ads: как не получить дубли

Есть два рабочих пути:

- **A. Импорт конверсий из GA4 в Google Ads** (часто проще).
- **B. Прямой Google Ads conversion tag** через GTM.

Рекомендация:

- Выбрать один путь как primary для одной конверсии.
- Если используешь оба, исключить дубль (разные conversion actions или четкие правила дедупликации).

---

## 5) Обязательная передача параметров между доменами

При переходе с лендинга в приложение важно сохранять:

- `utm_source`, `utm_medium`, `utm_campaign`, `utm_content`, `utm_term`
- `gclid` (Google Ads)
- опционально `fbclid`, `ttclid` и пр.

Практика:

- Не обрезать query-параметры при редиректах.
- Если есть промежуточные роуты, прокидывать параметры дальше.

---

## 6) События воронки (минимальный стандарт)

### Лендинг

- `page_view`
- `cta_click` (например, "Start free", "Try now")
- `pricing_view`

### Приложение

- `signup_started`
- `signup_completed`
- `login`
- `project_created`
- `thumbnail_generated`

Для всех ключевых событий держать единый нейминг и параметры (например, `source_domain`, `flow_step`, `plan_intent`).

---

## 7) Сквозная атрибуция (рекомендуется)

Помимо GA4, полезно хранить атрибуцию у себя (backend/DB):

- First touch (`utm_*`, timestamp, landing page).
- Last touch.
- Привязка к `user_id` после регистрации.

Это помогает сверять данные между GA4/Ads и продуктовой аналитикой.

---

## 8) Проверка перед релизом (чеклист)

- GTM установлен на обоих доменах.
- В GA4 есть события с обоих доменов в одном property.
- Переход между доменами не стартует "нового" пользователя в DebugView (в большинстве сценариев).
- `gclid` и `utm_*` сохраняются после перехода.
- Конверсия регистрации срабатывает ровно 1 раз.
- Конверсия импорта в Ads видна в статусе "recording conversions".

---

## 9) Частые ошибки

- Разные GA4 property для лендинга и приложения.
- Разные GTM контейнеры без синхронизации логики событий.
- Потеря query-параметров на редиректах.
- Одновременный импорт в Ads из GA4 и прямой Ads tag без контроля дублей.

---

## 10) Рекомендуемый порядок внедрения

1. Единый GTM на оба домена.
2. Единый GA4 + cross-domain.
3. События воронки (лендинг + app).
4. Проверка в DebugView.
5. Подключение конверсий Google Ads.
6. Дедупликация и финальная QA-проверка.
