# Unit-экономика и кредитная модель (MVP)

Документ фиксирует, как сейчас считается стоимость генерации в кредитах, где есть риски по марже, и какие ручки можно крутить при следующем пересчете.

---

## 1) Текущее состояние в коде

## Формулы списаний

### `project generate` (`POST /api/projects/:id/generate`)
- Списание: `1 кредит` за `1` вариант (`thumbnail variant`).
- Если часть вариантов упала, выполняется `refund` за недогенерированные.

### `pipeline run` / `pipeline run-video`
- Формула:
  - `1` кредит за анализ (VL/text step),
  - `+ N` кредитов за генерацию `N` картинок,
  - `+ 1` кредит за edit-step (если используется).
- Итого: `1 + N (+1 при edit)`.
- При ошибках/частичном успехе включен `refund` (резерв -> фактический расход).

## Пакеты кредитов (frontend pricing config)
- `100` кредитов за `$19`
- `300` кредитов за `$49`
- `700` кредитов за `$99`

Эффективная цена кредита:
- `$19 / 100 = $0.19`
- `$49 / 300 ≈ $0.163`
- `$99 / 700 ≈ $0.141`

---

## 2) Что это означает для MVP

Базово модель рабочая:
- простой mental model для пользователя;
- предсказуемое списание;
- есть компенсация (refund) при фейлах.

Для запуска этого обычно достаточно, но есть важные ограничения по unit-экономике (см. ниже).

---

## 3) Риски по марже

## Риск A: `premium` модель не стоит дороже в кредитах

Сейчас `default` и `premium` tier тарифицируются одинаково, хотя фактическая себестоимость `premium` обычно выше.

Последствие:
- чем больше пользователи выбирают premium, тем выше риск снижения валовой маржи.

## Риск B: фиксированный `1 кредит` за анализ

Анализ видео по факту может отличаться по себестоимости:
- URL vs upload,
- удачный frame path vs fallback,
- retries/fallback model path.

При этом цена анализа фиксирована и не учитывает сложность.

## Риск C: raw `video_url` в VL-модель может быть слишком дорогим

По тестам OpenRouter за 1 час:
- `Gemini 2.5 Flash`: примерно `$0.43` за `3` запроса и около `1.59M` tokens;
- `FLUX.2 Pro`: примерно `$0.09` за `3` image generation запроса.

Вывод: основной COGS bottleneck — не генерация картинок, а видео/VL-анализ. Поэтому для MVP важно держать video analysis строго ограниченным.

---

## 3.1) Cost guardrails для video analysis

После теста с высокими расходами введены guardrails:

- analyze window: `240 секунд` (4 минуты) вместо 10 минут;
- sampled frames: `4` вместо 8;
- max frame width: `640px`;
- JPEG quality снижен для уменьшения payload;
- minimum usable frames: `2`;
- raw `video_url` fallback в Gemini/OpenRouter отключен для MVP.

Если кадры не удалось извлечь:
- VL получает текстовый контекст, metadata и transcript snippet (если есть);
- raw video не отправляется в модель, чтобы не получить непредсказуемые video tokens.

Это может немного ухудшить анализ редких видео, где ffmpeg не смог достать кадры, зато защищает маржу и делает COGS предсказуемее.

---

## 4) Как пересчитывать после накопления данных

После первых продаж (и телеметрии по расходам) делаем пересчет:

1. Считаем фактический `COGS` по сценариям:
   - `project generate`
   - `pipeline run`
   - `run-video`
2. Считаем `COGS` отдельно по tier:
   - `default`
   - `premium`
3. Сверяем с текущей выручкой на 1 кредит по каждому пакету.
4. Проверяем целевую валовую маржу (например, 70%+ на MVP).

---

## 5) Рычаги корректировки (в порядке простоты)

## Вариант 1 — увеличить цену premium в кредитах
- Добавить коэффициент/надбавку для `image_model_tier=premium`.
- Пример: `premium` = `default + 1` кредит за каждую сгенерированную картинку.

## Вариант 2 — скорректировать наполнение пакетов
- Оставить `$`-цены как есть, но уменьшить количество кредитов в пакетах.
- Это самый прямой путь, если не хочется менять UX списаний внутри продукта.

## Вариант 3 — повысить базовую стоимость `run-video`
- Добавить маленький фикс за видео-пайплайн (например, +1 кредит).
- Использовать только если видно, что именно видео-флоу “съедает” маржу.

## Вариант 4 — снизить дефолтный `variant_count`
- Меньше картинок на запуск -> меньше COGS на пользователя.
- Можно вводить как продуктовый дефолт без изменения формулы кредитов.

---

## 6) Минимальный план на следующий этап

1. Запускаем текущую модель с cost guardrails.
2. Снимаем фактические затраты по ключевым сценариям.
3. Принимаем решение:
   - либо уменьшаем credits в пакетах,
   - либо вводим premium-множитель,
   - либо делаем оба шага умеренно.
4. Обновляем этот документ и публичные тексты на странице `Credits`.

---

## 7) Принцип принятия решений

На стадии валидации идеи важнее:
- простая и понятная пользователю механика,
- контроль кассового риска.

Поэтому сначала фиксируем продажи и поведение, затем пересчитываем модель на реальных данных, а не на предположениях.
